# AI開発診断 資料出し分けルール

## 質問フロー

診断は以下の7問で構成されています。

1. 作りたいものは？
2. 現在の状況は？
3. 想定ユーザー数は？
4. 必要機能数は？
5. ログイン機能は？
6. 決済機能は？
7. 開発希望時期は？

## 基本方針

添付するプレゼン資料PDFは、1問目の「作りたいもの」で決めます。

残りの6問は、資料そのものを切り替えるのではなく、診断結果PDF・メール本文・相談時の補足に反映します。

- 現在の状況: 提案文の温度感を変える
- 想定ユーザー数: 費用・期間・運用設計の注意点を変える
- 必要機能数: MVPか本格開発かを変える
- ログイン機能: 権限管理の有無を変える
- 決済機能: Stripe・返金・決済後運用の注意点を追加する
- 開発希望時期: 優先度とロードマップの表現を変える

## 補正ルール

### stage: 現在の状況

| 値 | 表示 | 使い方 |
| --- | --- | --- |
| idea | アイデア段階 | 作り込む前に仮説検証する文脈を強める |
| requirements | 要件整理中 | MVPスコープ確定の文脈を強める |
| designed | 設計済み | 実装・公開に向けた整理を強める |
| developing | 開発中 | 既存資産の引き継ぎ・改善を確認する |
| running | 運用中 | 改善、追加開発、リプレイスを確認する |

### users: 想定ユーザー数

| 値 | 表示 | 使い方 |
| --- | --- | --- |
| u100 | 100人未満 | 標準MVPとして扱う |
| u1k | 100〜1,000人 | 標準MVPとして扱う |
| u10k | 1,000〜10,000人 | 負荷・運用設計の注意を追加する |
| u10kup | 10,000人以上 | 本格SaaS・高負荷開発として個別相談を促す |

### features: 必要機能数

| 値 | 表示 | 使い方 |
| --- | --- | --- |
| f3 | 3個以下 | 初期スコープを3機能に絞る |
| f5 | 5個程度 | 初期スコープを5機能にする |
| f10 | 10個程度 | 段階開発・本格開発の注意を追加する |
| fNA | 未定 | 要件整理から入る |

### auth: ログイン機能

| 値 | 表示 | 使い方 |
| --- | --- | --- |
| yes | 必要 | ログイン、権限、管理者設定をスコープに含める |
| no | 不要 | 公開画面中心の軽いMVPとして扱う |

### payment: 決済機能

| 値 | 表示 | 使い方 |
| --- | --- | --- |
| yes | 必要 | Stripe、返金、決済後メール、注文管理の注意を追加する |
| no | 不要 | 決済なしのMVPとして扱う |

### schedule: 開発希望時期

| 値 | 表示 | 使い方 |
| --- | --- | --- |
| asap | 最短での開始を希望 | 優先度を高くし、早期要件整理を促す |
| m1 | 1ヶ月以内 | 初回相談からスコープ確定までを急ぐ |
| m3 | 3ヶ月以内 | 段階的な準備スケジュールを提示する |
| na | 時期は未定 | 要件整理・検証相談として扱う |

## 実装メモ

メール添付の基本は以下の2段構成がよいです。

1. 診断回答から生成する個別PDF
   - 費用、期間、推奨技術構成、回答概要を反映する

2. productに対応した横型プレゼンPDF
   - 上記マッピング表のPDFを添付またはリンクで案内する

メールサイズを抑える場合は、個別PDFだけ添付し、横型プレゼンPDFはリンクで案内します。
